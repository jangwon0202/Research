


**(1)셰이딩(Shading)** - 빛과 그림자, 색을 계산해서 입히는 모든 과정

**(2)컬링(Culling)** - 화면에 안 보이는 쓸데없는 걸 골라내서 버리는 작업.

**(3)클러스터(Cluster)** - 연관 있는 것들끼리 묶어놓은 덩어리


**(4)비저블 클러스터** - 화면에 실제로 보일 가능성이 있는 3차원 공간이나 기하학(Geometry)의 덩어리(Cluster)

**(5)클러스터드 셰이딩(Clustered Shading)** - 게임 화면을 3차원 상자 덩어리(클러스터)들로 나누고 각 구역(덩어리)마다 필요한 조명만 골라서 칠해지는 기술

**(6)메시 컬링(Mesh Culling)** - 그물망(Mesh)처럼 얽혀 있는 복잡한 3D 오브젝트 중에서 카메라에 안 보이는 부분은 과감하게 솎아내서(Culling) 렌더링 계산에서 제외하는 기술

**(7)프러스텀 컬링(Frustum Culling)** - 카메라의 시야 범위(절두체 모양) 밖에 있는 오브젝트들은 그래픽 카드가 아예 계산을 안하도록 하는 기술

**(8)오클루전 컬링(Occlusion Culling)** - 시야 범위 안에 있더라도, 큰 건물이나 벽에 가려져서 '실제로 눈에 안 보이는 뒤쪽 사물'을 판별해서 그리기를 생략하는 기술

**(9)LOD(Level of Detail)** - 카메라와 가까운 사물은 정밀하게(ex:수만 개의 삼각형), 멀리 있는 사물은 대충(ex:수십 개의 삼각형) 그리도록 실시간으로 메시를 교체하는 기술

**(10)메시(Mesh)** - 입체적인 물체(Object)를 수많은 점과 선, 다각형 등으로 구성한 그물망 형태의 3D 모델 데이터

**(11)SSR(Screen Space Reflection)** - 화면에 찍힌 정보만을 활용해 도로 위의 물웅덩이나 차량 표면에 주변 환경을 실시간으로 반사하는 기법

**(12)Compute Shader 기반 센서 모방** - 가상의 레이저를 수만 발 쏘는 라이다(LiDAR)나 초음파 센서의 물리적 충돌 연산을 GPU의 병렬 연산(Compute Shader)으로 처리하여 데이터화하는 기술.

**(13)깊이/열화상 버퍼 렌더링** - 화면을 일반 색상이 아닌, 카메라와의 거리를 기반으로 한 깊이 값(Depth)이나 적외선 센서용 흑백/열화상 텍스처르 뽑아내는 멀티 렌더 타겟(MRT) 기술

**(14)카메라 매칭 (Matrix Synchronization):** 실제 촬영 현장의 카메라 위치/초점 거리 데이터를 실시간으로 받아와, 언리얼 엔진 속 가상 카메라의 View Matrix와 Projection Matrix에 1ms의 오차도 없이 실시간 동기화하는 기술.

**(15)Genlock (제너레이터 락):** 실제 카메라의 셔터 타이밍과 LED 스크린의 주사율(프레임)을 완벽하게 일치시켜, 화면이 지직거리거나 검은 줄(롤링 셔터 현상)이 가는 걸 막는 하드웨어 연동 기술.

**(16)ACES (Academy Color Encoding System):** 영화 산업 표준 색공간. 모니터, LED, 카메라 기기마다 제각각인 색 표현을 수학적으로 통일하는 컬러 파이프라인 구축 기술.

**(17)레이 트레이싱 (Ray Tracing):** 빛의 알갱이가 튕겨 나가는 경로를 역추적하여, 유리창의 굴절과 정밀한 그림자를 물리적으로 완벽하게 시뮬레이션하는 고급 렌더링 기술.

**(18)볼륨 렌더링 (Volume Rendering)**
	- **(1)레이 마칭 (Ray Marching / Ray Casting):** 일반적인 3D 캐릭터는 속이 텅 빈 삼각형 껍데기지만, CT 데이터는 속이 꽉 차 있어. 카메라에서 빛을 발사해 이 3D 데이터 볼륨 내부를 통과시키면서, 밀도에 따라 색을 누적해 정밀한 내부 장기를 표현하는 기술.
	  - **(2)등가면 추출 (Isosurface Extraction / Marching Cubes):** 3D 밀도 데이터에서 "뼈의 밀도에 해당하는 부분만 골라내서 삼각형 메쉬 모양 껍데기로 자동 변환해라" 할 때 쓰는 알고리즘.

**(19)화면 찢어짐과 주사율 제어 부문**
	**(1)수직 동기화 (V-Sync)**
		- GPU의 프레임 생성 속도를 **모니터의 주사율(60Hz, 144Hz 등)에 강제로 동기화**시켜서 딱 맞춰 렌더링하는 기술.
		- 장점 - 화면 찢어짐이 사라짐. GPU가 과열되지 않음.
		- 단점 - GPU가 모니터 신호를 기다리기 때문에, 키보드나 마우스를 입력했을 때 화면에 반응되는 속도가 밀리는 인풋 렉(입력 지연)이나 프레임이 뚝뚝 끊기는 스타터링이 생길 수 있음.
	**(2)삼중 버퍼링(Triple Buffering)**
		- GPU 내부에 화면을 그리는 도화지(버퍼)를 기본 2개(전면/후면)에서 3개로 늘리는 기술
		- V-sync를 켰을 때 발생하는 프레임 저하와 대기 시간을 줄이기 위해, GPU가 쉬지 않고 세 번째 버퍼에 미리 다음 화면을 그리게 만들어 인풋 렉을 다소 완화해주지만 GPU 메모리(VRAM)을 조금 더 먹게 하는 단점이 존재.

**(20)픽셀 외곽선 깨짐(안티앨리어싱 제어)**
	**(1)TAA(Temporal Anti-Aliasing) / TSR(Temporal Super Resolution)**
		- 시간(Temporal) 개념을 도입해서, 바로 이전 프레임들의 화면 정보들을 프레임 버퍼에서 가중치를 두어 합성해 테두리를 부드럽게 깎는 기술
		+ **TAA**는 언리얼 4와 5의 기본 안티앨리어싱 기술로 정지 화면에서는 부드럽지만, 화면을 급격하게 돌릴 때 화면이 살짝 뿌옇게 번지는 현상이 있음.
		   이를 더 발전시킨 게 언리얼 5의 최신 업스케일러 겸 안티앨리어싱 기법인 **TSR**이다
	**(2)FXAA(Fast Approximate Anti-Aliasing)**
		-수학적 연산 없이, 다 그려진 최종 2D 화면에서 계단 모양이다 싶은 픽셀 경계선만 포토샵 블러(Blur) 필터처럼 빠르게 문지르는 기술
	
**(21)빛과 그림자, 그리고 디테일**
	**(1)포스트 프로세싱 (Post Processing, 후처리)**
		- GPU가 3D 연산을 다 끝내고 화면을 모니터에 출력하기 직전, 최종 2D 이미지 상태 위에 필름 효과나 카메라 렌즈 효과 등을 덧입히는 가공 단계.
		- 블룸(Bloom) : 전구나 태양 같은 밝은 빛 주변으로 빛이 은은하게 번지는 효과
		- 모션 블러(Motion Blur) : 카메라를 빠르게 돌릴 때 속도감 때문에 화면이 흐려지는 효과
		- 비네팅(Vignetting) : 화면 구석탱이 외곽 부분을 어둡게 깎아서 영화 같은 몰입감을 줌.
	**(2)스크린 스페이스 앰비언트 오클루전(SSAO/GDAO)**
		- 사물과 사물이 만나는 구석진 틈새나 골짜기 영역에 실시간으로 미세한 음영(그림자)을 계산해 넣는 기술
		+ 그림자가 생겨야 할 부분에 자연스럽게 검은색 음영이 생기면서 깊이감과 몰입감을 높여줌
	**(3)텍스처 필터링(Anisotropic Filtering, 비등방성 필터링)**
		- **텍스처 무늬가 일그러지거나 흐려지지 않고 끝까지 선명하게** 보이도록 보정하는 기술
**(22)컴퓨터 구조론 관점의 그래픽스 용어**
	**(1)렌더 타깃 (Render Target)**
	- GPU가 연산한 화면 데이터를 모니터에 바로 쏘지 않고, 메모리(VRAM) 안의 임시 텍스처 공간에 먼저 그려두는 가상의 도화지
	**(2)드로우 콜 (Draw Call) / 드로우 인덱스드 (Draw Indexed)** **(⭐중요!!⭐))**
		- CPU가 GPU에게 화면에 무엇을(Mesh), 어떻게(Material) 그릴지 지시하는 명령.
		- 게임 엔진이 그래픽 API(DirectX, OpenGL 등)를 통해 렌더링 파이프라인으로 보내는 신호
		**(1)드로우 콜이 많아지면 생기는 문제**
		- **CPU 병목 현상** : 드로우 콜을 준비하는 과정은 CPU에서 이루어지는데 한 프레임에서 드로우 콜이 너무 많으면 CPU 과부하가 걸려 프레임 드랍이 발생(CPU Bound 병목)
		- **성능 저하** : GPU의 실제 연산 성능이 남아있더라도 CPU가 명령을 따라가지 못해 렌더링 효율이 떨어짐
		**(2)드로우 콜을 최소화하는 법**
			드로우 콜을 합치려면 반드시 오브젝트들이 **동일한 머티리얼(텍스처 및 셰이더)**을 공유해야 함.** 재질이 다르면 하드웨어적으로 드로우 콜을 묶을 수 없음
			**(1)배칭(Batching)**
				- 여러 개의 오브젝트를 하나로 묶어서 한 번에 그리는 기술
			**(2)정적 배칭(Static Batching)**
				- 크기 변환(Transform)이 없고 움직이지 않는 배경 오브젝트를 하나로 결합
				- 실무 응용 - 여러 배경 프롭들이 각자 다른 텍스처를 쓰면 배칭이 깨짐. 그러므로 여러 이미지를 큰 메쉬로 합치는 텍스처 아틀라스(Texture Atlas) 작업이 필수적으로 동반되어야 함.
			**(3)동적 배칭(Dynamic Batching)**
				- 움직이는 작은 오브젝트의 정점 데이터들을 CPU에서 매 프레임마다 실시간으로 연산해 하나의 버퍼로 미리 합쳐 GPU에 전달하여 렌더링함
				- 정점 수가 아주 적은 메쉬만 효과가 있으며 정점 수가 많으면 오히려 CPU가 정점 합산 연산을 하느라 새로운 렉을 유발할 수 있음.
			**(4)GPU 인스턴싱(Instancing)**
				**개념** - 같은 메시와 머티리얼을 사용하는 수많은 오브젝트를 단 한번의 드로우 콜로 처리하는 기술
				**원리** - 메쉬 데이터는 VRAM에 딱 1개만 로드하고, 각 오브젝트들의 다른 위치 변환(Transform) 값만 인스턴스 버퍼(배열)로 묶어 GPU에 전달해 하드웨어 전역에서 대량 양산함.
		**(3)왜 프레임 드랍이 일어날까??**
		- 드로우 콜을 보낼 때 2가지 하드웨어 오버헤드가 발생하기 때문.
			**(1)유저 모드 -> 커널 모드 전환 오버헤드**
				 **(1) 매커니즘**
				 - CPU가 그래픽 카드(하드웨어)를 제어하려면 유저 레벨 응용 프로그램(게임 엔진)에서 OS 커널 레벨(그래픽 드라이버)로 권한을 넘기는 시스템 콜(System Call)이 발생해야 함.
				**(2) 병목 원인**
				- 유저 모드에서 커널 모드로 전환될 때마다 CPU는 현재 프로세스의 CPU 레지스터 상태를 안전한 곳에 대피시키고 커널 코드를 로드하는 등, CPU 스레드 자체에 연산 비용(오버헤드)이 크게 발생
				**(3) 결과**
				- 드로우 콜이 수만 번 발생한다 = CPU가 1프레임(0.016s) 안에 하드웨어 시스템 콜을 수만 번 날린다는 뜻.
				  CPU 렌더 스레드가 이 모드 전환 오버헤드를 감당하지 못해 점유율 100%를 찍고 과부하가 걸림.
			**(2)GPU의 컨텍스트 스위칭(상태 변경) 오버헤드**
				**(1) 매커니즘**
				- 드로우 콜과 드로우 콜 사이에 물체의 텍스처, 셰이더 변수, 블렌드 모드 등이 바뀌면 GPU는 내부 연산 유닛들의 하드웨어 셋팅을 싹 갈아엎는 상태 변경(State Change / 레지스터 재할당)을 수행해야 함.
				**(2) 병목 원인**
				- GPU는 대규모 병렬 연산(SIMD) 구조이기 때문에 한번 세팅된 사격 제원으로 수만 개 픽셀을 동시에 쏘아야 효율이 극대화됨. 중간에 재질이 바뀌어 세팅을 새로 고치는 동안 GPU 내부의 수천 개 코어들은 대기 상태에 빠짐.
				**(3) 결과**
				- 이전 드로우 콜의 연산이 완전히 완료(Flush)되고 새 세팅이 끝날 때 까지 연산 유닛들이 멍하니 노는 파이프라인 스톨(Pipeline Stall, 가동 중단) 현상이 누적되어 화면을 제때 그리지 못하고 프레임 드랍이 발생.
				

**(23)프리미티브(Primitive)** : 가장 원시적인 기본 도형
