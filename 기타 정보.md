1.**절차지향(Procedural Programming)**
- 실행되는 순서와 절차를 중요하게 생각하는 방식.
   데이터보다 "무엇을 할 것인가?"라는 함수 중심의 설계
   **위에서 아래로 논리적이고 절차적인 순서가 핵심**
ex) C언어
장점 - 실행 속도가 빠르고 컴퓨터의 처리 방식과 유사
단점 - 데이터와 함수가 분리되어 있어, 규모가 커지면 유지보수가 매우 어려움(하나를 고칠 때 전체가 꼬일 수 있음)

2.**객체지향(Object-Oriented Programming)**
- 프로그램을 여러 개의 독립된 **객체**들의 모임으로 보는 방식. 데이터와 그 데이터를 처리하는 함수를 하나로 묶은 객체 중심의 설계.
**'클래스'라는 설계도를 통해 객체를 찍어낸 후, 객체끼리 서로 상호작용하게 한다.**
ex) Java, C++, C# 등

비유(자동차 만들기) - 엔진객체, 바퀴객체, 핸들객체 등을 따로 만들어 조립함

장점:코드의 재사용성이 높고 부품(객체)만 갈아끼우면 되기 때문에 유지보수가 쉬움
단점:절차지향에 비해 설계 시간이 오래 걸리고 실행 속도가 상대적으로 느리고 디버깅이 복잡함

3.**함수형 프로그래밍(Functinal Programming)**
->'무엇을 실행할까'보다 '**함수(수학적 함수)를 어떻게 조합할까**'에 집중하는 방식
   데이터의 상태를 바꾸지 않고(불변성), 입력값이 같으면 항상 같은 결과(순수 함수)를 내놓는 것을 목표로 함.

비유:컨베이어 벨트에 재료를 넣으면 정해진 공정(함수)를 거쳐 결과물이 나오는 공장라인과 같음

장점:프로그램의 동작을 예측하기 쉽고, 여러 작업을 동시에 처리(병렬 처리)할 때 오류가 적음

언어:Haskell, Lisp이 대표적이며 JavaScript, Python, C# 등에서도 쓰임

결론
(1)명령형(How): "문 열고, 냉장고 앞으로 가서, 우유를 꺼내와"처럼 과정 하나하나를 컴퓨터에게 상세하게 명령함(절차지향,객체지향)

(2)선언형(What): "우유 한 잔 줘."처럼 내부적으로 어떻게 가져오는지는 신경쓰지 않고, 내가 얻고 싶은 결과가 무엇인지만 정의함(함수형,SQL 등)

C#이나 C++도 근본적으로는 객체지향 언어지만 함수형 프로그래밍의 요소(C#의 LINQ, C++의 Lambda 등) 굉장히 많이 섞어서 사용하며 이를 **멀티 패러다임 언어**라고 부름.

**4.네트워크 토폴로지**
	**(1)물리적 토폴로지**
- 여러 대의 컴퓨터와 장비들이 물리적으로, 논리적으로 어떤 모양으로 연결되어 있는지 배치 방식을 의미함
  
	주요 토폴로지 유형
	- (1)스타형(Star) - 중앙 허브에 모든 장치가 연결됨
		장점 - 관리가 쉽고 한 곳이 고장나도 나머지는 안전함
		단점 - 중앙 허브(서버)가 터지면 전체 네트워크가 마비됨
	- (2)버스형(Bus) - 하나의 공통 간선에 장치들이 붙어있음
		장점 - 설치 비용이 저렴하고 간단함
		단점 - 메인 케이블에 문제가 생기면 전체 네트워크가 마비됨
	- (3)링형(Ring) - 양옆의 장치와 고리 모양으로 연결
		장점 - 충돌이 적고 데이터 흐름이 일정함
		단점 - 노드 하나만 고장나도 전체 네트워크가 마비됨
	- (4)망형(Mesh) - 모든 장치가 서로 그물처럼 연결
		장점 - 신뢰성이 매우 높고 우회경로가 많음
		단점 - 설치 비용이 비싸고 구조가 복잡함
	- (5)트리형(Tree) - 계층형 구조(스타형의 확장)
		장점 - 확장이 쉽고 대규모 네트워크에 적합함
		단점 - 상위 노드에 장애가 발생 시 하위 노드 전체에 모두 영향

**(2)논리적 토폴로지** 
- 데이터가 누구를 거쳐서 가는가?
  
	(1)클라이언트-서버 아키텍처 (Client-Server / Star형 기반)
	- 중앙에 서버(Host)가 있고, 모든 플레이어(Client)가 네트워크로 서버에 접속하는 1:N 구조
	  특징:데이터의 무결성,보안,업데이트가 서버 한 곳에서 이루어짐
		
		장점
		- 강력한 보안 - 서버에서 인증과 권한 관리를 중앙 제어하므로 보안 사고 대응이 빠름
		- 데이터 일관성 - 모든 데이터가 중앙 DB에 저장되므로, 어떤 클라이언트에서 접속해도 동일한 상태를 유지
		- 백업 및 유지보수 용이 - 서버 한 곳만 관리하면 되므로 데이터 백업이나 시스템 업데이트가 효율적임
		  
		단점
		- 단일 실패 지점 - 서버가 고장 나거나 트래픽이 몰려 터지면 모든 클라이언트가 서비스 이용이 불가능함
		- 고비용 - 고성능 서버 장비 구매와 운영(데이터 센터, 전기세, 회선비 등)에 막대한 비용이 든다
		- 네트워크 병목 - 접속자가 급증하면 중앙 서버에 부하가 집중되어 속도가 느려짐


	(2)P2P 아키텍처 (Peer-to-Peer)
	- 중앙 서버 없이 플레이어끼리 직접 데이터를 주고받음
	- 각 컴퓨터의 CPU,저장공간,대역폭이 모여 전체 네트워크의 자원이 됨
	  
		장점
		- 확장성 - 사용자가 많아질수록 데이터를 공유하는 공급자가 많아지므로, 이론적으로 네트워크 전체 성능이 좋아짐
		- 비용 절감 - 중앙 서버를 운영할 필요가 없어 인프라 비용이 거의 들지 않음
		- 높은 가용성 - 특정 컴퓨터 몇 대가 고장 나도 전체 네트워크는 여전히 동작함
		  
		단점
		- 보안 취약 - 각 컴퓨터끼리 직접 연결되므로 IP 노출 위험이 있고, 악성 데이터를 퍼뜨리는 노드를 제어하기 어려움
		- 데이터 일관성 부족 - 누가 어떤 데이터를 가지고 있는지 실시간으로 파악하기 어렵고, 데이터가 유실될 가능성이 높음
		- 성능 불균형 - 인터넷 속도가 느린 사용자가 포함되면 전체 네트워크 속도에 영향을 줄 수 있음(가장 느린 사람에게 맞춰지기 때문)
		  
	+ 로그인이나 결제처럼 보안이 중요한 영역은 클라이언트-서버로 처리
	  대용량 게임 파일 다운로드나 1:1 보이스 채팅 등은 비용 절감을 위해 P2P 기술을 섞어씀
	  
	  
**5.서버**
	(1)데디케이티드 서버(Dedicated Server) - 그래픽 화면 없이 게임 로직만 계산하는 전용 서버 프로그램
		 장점 - 모든 플레이어가 공평한 지연 시간(Ping)을 가짐. 서버가 매우 안정적
		 단점 - 서버 유지비(AWS 등)가 발생하며 설정이 복잡함.
		 ex)배틀그라운드, 리그 오브 레전드, 오버워치 등(대규모 게임)
	(2) 리슨 서버(Listen Server/ Host) - 방장을 맡은 플레이어의 PC가 게임도 돌리고 서버 역할도 함
		 장점 - 서버 운영 비용이 0원. 별도의 서버 구축 없이 바로 플레이가 가능
		 단점 - 방장의 컴퓨터가 느리면 모두가 렉이 걸림. 방장이 나가면 게임이 종료됨
		 ex)피코 파크, IT TAKES TWO(협동형)
